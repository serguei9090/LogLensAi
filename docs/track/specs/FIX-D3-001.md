# FIX-D3-001: Persistence & UI Polish for Workspace Engine Settings

## Issue Description
1. **Persistence Failure**: Workspace-specific Drain3 configuration (especially masking rules) fails to persist effectively across modal open/close cycles despite a successful save toast. Reopening the modal often reverts to global defaults.
2. **UI Regression**: The "Save Overrides" button in the `WorkspaceEngineSettings` modal lacks proper bottom padding, leading to a cramped and unproffessional appearance.

## Root Cause Analysis
- **Synchronization Gap**: The `WorkspaceEngineSettings` modal was relying on the shared store's `settings` state, which may not have been refreshed specifically for the workspace context upon opening. While `InvestigationPage` performs a fetch, race conditions or store clobbering (e.g., from navigating to Settings) could leave it in a stale "global" state.
- **Atomic Layout Mismatch**: The `DialogFooter` was using `p-4`, while the `DialogHeader` and content blocks used `p-6`, creating visual inconsistency and insufficient breathing room at the bottom of the viewport.

## Optimal Solution
1. **Force Sync**: Implement an explicit `fetchSettings(workspaceId)` call within the `WorkspaceEngineSettings` component's mount effect (on `isOpen`). This ensures the single source of truth (the Sidecar DB) is always queried right before the user begins editing.
2. **Padding Harmonization**: Update `DialogFooter` to use `p-6 pt-4` (or `px-6 pb-6 pt-4`) to align with the rest of the modal's Atomic Design spacing.

## Implementation Details
- Modified `WorkspaceEngineSettings.tsx` to trigger a store fetch on modal open.
- Refined `DialogFooter` className for better visual balance.

## Quality Guardrail
- Verify that saving a mask in Workspace A does not affect Workspace B.
- Verify that closing and reopening the modal in Workspace A correctly displays the saved mask.
- Verify visual padding at the bottom of the modal on various screen sizes.
